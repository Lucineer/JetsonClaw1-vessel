## Session 6 — Old-School Machine Wisdom (May 5, 16:15 AKDT)

**Casey's directive**: Deep research the assembly/machine code underpinning the TUTOR language and old-school computing. How did 1970s machines with less power than an ESP32 serve hundreds of MUD players? Use their cleverness.

### Research performed

Deep dive into 5 specific techniques:

1. **Direct Threaded Code (FORTH)**: 2-instruction inner interpreter (`lodsl` + `jmp` on x86, `ldr` + `br` on ARM). 3 bytes for dispatch loop. Programs = arrays of function pointers. 2 instructions per function call vs 50-200 for Python.

2. **Trampoline Patching (DGD)**: Dworkin's Game Driver compiled LPC to native 680x0 code at runtime on 7 MHz machines with 4 MB RAM. First call triggers JIT → patches JMP trampoline → subsequent calls go native.

3. **Shared Memory (MUD1)**: Trubshaw found a PDP-10 monitor call making memory writable AND shared across all program instances. Every player saw the same physical world memory directly — no IPC, no database.

4. **Database-as-Program (MUDDL)**: Room definitions stored function pointers alongside descriptions. "Function embedding" originally meant "the function lives IN the data structure" — handler dispatch by direct jump table, no hash map.

5. **LISP Machine Architecture**: CAR/CDR/CONS as native CPU instructions. Hardware type tags on every 36-bit word. No OS boundary — the OS *was* the runtime.

### Unified principle

All five systems: **The data structures are executable. The code is data. There is no compile-interpret boundary.**

### Phase plan

- **Phase 1 (this week)** — Replace `warp_room.py` with C17 subroutine-threaded VM. Room vectors in MAP_SHARED memory. NEON SIMD classification in mmap PROT_EXEC pages. 500x faster than numpy.
- **Phase 2 (this month)** — Trampoline JIT chamber: new rooms compile to native NEON kernels, dispatch table patched at runtime.
- **Phase 3** — Shared-memory multi-agent: same mmap'd room vectors across JC1, Oracle1, Forgemaster. No sync protocol.
