## 08:53 AKDT — Bench Push Session (Suites 52-57)

Casey said "continue" then "go go go" — ran 6 benchmark suites back to back.

### Suite #52: Sparse Kernel — Skip Zero Weights
- Dense wins below ~87% sparsity — random index lookups destroy L2 cache
- Cooperative sparse v2: 1.16× at 99% sparsity (best case)
- Sparse crossover at ~87% — only then does skipping zeros pay off
- Memory savings are the real win: 32× less at 99% sparsity
- Rule #37: Use dense unless >90% sparse

### Suite #53: Room Capacity & Memory Scaling
- **5.75M rooms fit at dim=256** (2.8 GB) — memory is NOT the bottleneck
- 134M qps at 65K rooms, 83.3% GPU efficiency
- Variable-dim serving is **40× SLOWER** — cumulative offset loop is O(n²)
- Always pad to uniform dimension (fleet standard: dim=256)
- Sweet spot: 4K-16K rooms per batch (70-81% efficiency)
- Rules #38-39: Standardize dims, batch 4K-16K

### Suite #54: Concurrent Kernel Execution
- **Multi-tenant inference 2.61× FASTER** than solo — GPU scheduler merges workloads
- Light compute background IMPROVES inference 2.27× (warm GPU effect)
- D2D copies are catastrophic: 265× slower (memory bus saturation)
- Heavy compute steals SMs: 6.2× slower
- H2D copies: 5.7× slower on unified memory
- Rules #40-42: Batch all inference, avoid memory ops, light BG helps

### Suite #55: Double-Buffered Weight Pipeline
- Double-buffering only 3.2% speedup on unified memory (HW already overlaps)
- Weight upload dominates: 149μs upload vs 38μs inference (3.9×)
- Pure inference: 62.5M qps cached vs 5.5M qps with uploads (11.4× gap)
- Partial updates scale linearly: 1% update = 6.1× faster than full
- Rules #43-44: Cache weights on GPU, upload only deltas

### Suite #56: Power Modes & Clock Scaling (MILESTONE)
- **163.4M room-qps sustained** over 10 seconds — NEW ALL-TIME RECORD
- GPU running at only 306 MHz (30% of max 1020 MHz) — nvpmodel default
- Theoretical max at full clock: ~544M qps (3.3× headroom)
- **Zero outliers** — p99/p50 = 1.027 in 5000 samples (tightest ever)
- Thermal stable at 51.6°C, 48°C headroom to junction max
- Peak at 4096 rooms: 162.8M qps, efficiency drops at 8K+
- Rules #45-46: Run jetson_clocks for production, zero-outlier latency achievable
- **NOTE:** #include <fstream> causes silent segfault on this Jetson with nvcc + -O3. Use fopen() only.

### Suite #57: Per-Room Inputs
- Per-room input FASTER at small batches (<2000 rooms): 0.63× at 64 rooms
- Crossover at ~2000 rooms — L2 cache capacity limit (2MB)
- At 4096 rooms: per-room 2.42× slower (input array = 2MB, evicts weights)
- Rule #47: Use shared input when batch > L2 capacity

## Session 2: Suites 58-69 (Novel Techniques Sprint)

### Suite #58: BF16 vs FP16
- BF16 throughput = FP16, but accuracy 176% worse (8 mantissa bits vs 11)
- Rule #48: Use FP16, not BF16

### Suite #59: Launch Overhead & CUDA Graphs
- Launch overhead = 4.3μs (99.8% at 1 room)
- **CUDA Graphs are 4-9× SLOWER on Jetson** — opposite of data-center wisdom
- Rule #49-50: Batch accumulation, not graphs or persistent kernels

### Suite #60: Custom vs cuBLAS/cuSPARSE
- V7 custom kernel beats cuBLAS 22% and cuSPARSE 170% at 4K rooms
- CSR sparse format actually LARGER than dense at 50% sparsity
- Rule #51-52: Custom kernels win for small matrices

### Suite #61: Shared Memory Tiling
- SHM tiling 13-33% slower — L2 already provides equivalent locality
- Rule #53: Don't tile single-pass dot products into shared memory

### Suite #62: L2 Cache Persisting
- Partial persist (1K rooms) 23% faster than no persist
- FP16 accumulation 22% faster (but shfl_sync on half is broken)
- 1M sustained: 153.4M qps
- Rule #54-55: Persist only what fits in L2, use FP16 accum

### Suite #63: Warp Specialization & Cache Intrinsics
- `__ldg()` 41% slower — bypasses L1 for no benefit on sequential reads
- Warp specialization 100× slower — sync overhead kills single-pass ops
- LDP unroll4: 7-10% faster at 4K+ rooms
- Rule #56-57: Never use __ldg for sequential access, warp spec needs multi-stage

### Suite #64: Combined & Block Size Sweep
- "319M qps" was a correctness bug — V9 left rooms 8-15 unprocessed
- 128-thread blocks 14% faster than 256 (better SM occupancy)
- Rule #58-59: Verify correctness before speed, 128t blocks optimal

### Suite #65: INT8 Quantization
- **1.36× faster** than FP16, 4.15% avg error, 50% memory reduction
- Per-room symmetric quantization preserves ranking (99.2% top-256 recall)
- Rule #60: INT8 for room ranking

### Suite #66: Progressive Refinement
- INT8→FP16 refinement only 1.08× (sort overhead negates gain)
- INT8 alone is best strategy for ranking
- Rule #61: Skip progressive refinement

### Suite #68: Launch Bounds & Compiler Flags
- `__launch_bounds__(256,8)` = 20% free improvement
- `--use_fast_math` = additional 8%
- Rule #62-63: Always use launch bounds + fast_math

### Suite #69: Ultimate Combined
- **INT8 + launch_bounds + fast_math = 185M qps sustained**
- 43% improvement over untuned baseline, zero algorithm changes
- L2 persist adds nothing for INT8 (1MB fits in 1.4MB L2)
- Theoretical max at jetson_clocks: ~616M qps
- Rule #64: The production stack
