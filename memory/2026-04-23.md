## 18:05 AKDT — Tensor Core WMMA: SOLVED

### Bug Found & Fixed
Previous tensor core kernel only loaded first 16 input elements, reused across all 16 chunks. Each chunk was computing input[0:16] * weights[chunk*16:chunk*16+16] instead of input[chunk*16:chunk*16+16] * weights[chunk*16:chunk*16+16].

Fix: Load BOTH input AND weights into shared memory per chunk.

### WMMA API Reference (learned through debugging)
- `fragment<accumulator,16,16,16,float>.num_elements = 8` (NOT 256!)
- `cf.x[0]` on lane 0 = `C[0][0]` = exact 16-element dot product
- `load_matrix_sync(af, ptr, 16)` with row_major: fills first ROW of 16x16 matrix
- `load_matrix_sync(bf, ptr, 16)` with col_major: fills first COLUMN of 16x16 matrix
- `mma_sync(cf, af, bf, cf)` computes C = A*B where C[0][0] = Σ(a[i]*b[i])
- MUST load both A and B fragments per chunk when iterating over input chunks
- Fragment A CAN be reused across iterations (verified with known-value tests)

### Final Benchmark Results
- Thread (256 thr): 0.0085 ms, 117,821 qps
- Warp shuffle (32): 0.0060 ms, 166,004 qps (1.41x faster than thread)
- Tensor core WMMA: 0.0076 ms, 131,734 qps (0.79x vs warp, but CORRECT)
- Max absolute diff (warp vs tc): 5.01e-04 (FP16 precision)
- All 6 rooms: ✓ EXACT match

### Why Warp Still Wins
256-element dot product is memory-bound. Shared memory load + sync overhead
for WMMA dominates the small computation. Tensor cores will win with larger
matrices (e.g., room weight matrices in full inference pipeline).
