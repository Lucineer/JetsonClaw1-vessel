## Technical Discoveries

### GGUF v3 Metadata Format (critical fix)
```
metadata_entry:
  key: string (8-byte len + data)
  value_type: uint32
  value: variable (depends on type)
```
My original code read `value_type` as the first field and interpreted `key_len` bytes as value_type. Caused "key_len" of 7308890738324930560 and 0x14=20 as invalid type.

### CMA Depletion Root Cause
After the first ggml_init() call, the GPU driver allocated CMA for CUDA and never released it. From 344MB → ~20MB free. Now at 644KB. Only fix is reboot.

### ggml Pip Package CUDA Binding
The llama-cpp-python pip package compiles ggml with CUDA. Even `libggml.so` (the tiny backend loader) links to `libggml-cuda.so` which links CUDA libs. Can't extract a CUDA-free subset.

## Session 2 — flato MUD Skeleton (01:00 AKDT)

### Summary
Built the **flato MUD** — Fleet Plato MUD in C17. A telnet-based multi-agent MUD that routes prompts to edge-llama for inference. No dependencies, pure poll() event loop.

### Key Results
- ✅ **flato.c compiled** — 19KB binary, zero dependencies (`cc -std=c17 -O2`)
- ✅ **Pure poll() event loop** — handles telnet, edge-llama Unix socket, and clients in a single thread
- ✅ **Telnet server** — binds 0.0.0.0:4000, telnet banner + command prompt
- ✅ **Commands**: `/think`, `/help`, `/status`, `/peers`, `/quit`
- ✅ **edge-llama bridge** — connects via `/tmp/edge-llama.sock`, sends JSON requests, streams responses
- ✅ **Non-blocking I/O** — all sockets O_NONBLOCK, no threading
- ❌ **Can't test end-to-end** — edge-llama can't generate without CUDA (CMA depleted)

### flato Architecture
```
flato.c (17KB, pure C17)
  ├── Telnet listener (port 4000)
  ├── Client pool (max 64)
  ├── edge-llama bridge (Unix socket)
  └── Mesh peer list (UDP, future)
```

### Design Notes
- flato owns the connection pool and routing; edge-llama owns compute
- Session: `/think "what's the weather?"` → prompt to edge-llama → stream response to telnet client
- Context window tracked as `tokens[MAX_TOKEN_HIST]` per client
- Mesh peers will use UDP broadcast discovery (future)
