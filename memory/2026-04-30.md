## Technical Discoveries

### GGUF v3 Metadata Format (critical fix)
```
metadata_entry:
  key: string (8-byte len + data)
  value_type: uint32
  value: variable (depends on type)
```
My original code read `value_type` as the first field and interpreted `key_len` bytes as value_type. Caused "key_len" of 7308890738324930560 and 0x14=20 as invalid type.

### CMA Depletion Root Cause
After the first ggml_init() call, the GPU driver allocated CMA for CUDA and never released it. From 344MB → ~20MB free. Now at 644KB. Only fix is reboot.

### ggml Pip Package CUDA Binding
The llama-cpp-python pip package compiles ggml with CUDA. Even `libggml.so` (the tiny backend loader) links to `libggml-cuda.so` which links CUDA libs. Can't extract a CUDA-free subset.

### Streaming Architecture (C side — built prev session)
```
edge_generate_stream(model, prompt, max_tokens, out_len, new_tokens, callback, user_ctx)
  └─ do_generate(..., callback)  // shared refactored loop
       ├─ llama_decode
       ├─ llama_sampler_init_greedy / llama_sampler_sample
       ├─ llama_token_to_piece → callback(piece, len)
       └─ loop until max_tokens or EOS
```
The streaming callback fires for each token piece during generation, called from inside the `do_generate()` loop. The C mutex (`std::mutex g_gen_mutex`) ensures thread safety.

### Streaming Architecture (MUD side — today)
```
@infer "hello" ──▶ "🤖 " msg + return immediately
  └─ spawn Thread(target=_stream_inference, daemon=True)
       └─ _send_piece(token) → reactor.callFromThread(caller.msg, piece)
            └─ (main thread) caller.msg(piece) 
       └─ Done → reactor.callFromThread(caller.msg, "⚡ 18 t/s")
```

### Key Decision
- **Threaded streaming is the right approach for Evennia** — Evennia's Twisted reactor is single-threaded. Blocking inference in the command handler means messages queue but don't flush until command returns. Spawning a background thread and using `reactor.callFromThread` for `caller.msg()` lets tokens appear progressively. The C mutex serializes concurrent access when multiple users call @infer.

### Test Results
- Streaming C API tested directly via Python script: **18 t/s**, each token piece fires Python callback
- Model loaded successfully in Evennia process at server start
- @infer command registered in CharacterCmdSet

### Not Tested (trust reasoning)

Architecture is sound: `reactor.callFromThread` is the standard Twisted pattern for thread→main-thread communication. If Evennia's `caller.msg()` from the main thread flushes immediately, tokens will appear progressively.

### Streaming Architecture (C side — built prev session)
```
edge_generate_stream(model, prompt, max_tokens, out_len, new_tokens, callback, user_ctx)
  └─ do_generate(..., callback)  // shared refactored loop
       ├─ llama_decode
       ├─ llama_sampler_init_greedy / llama_sampler_sample
       ├─ llama_token_to_piece → callback(piece, len)
       └─ loop until max_tokens or EOS
```
The streaming callback fires for each token piece during generation, called from inside the `do_generate()` loop. The C mutex (`std::mutex g_gen_mutex`) ensures thread safety.

### Streaming Architecture (MUD side — today)
```
@infer "hello" ──▶ "🤖 " msg + return immediately
  └─ spawn Thread(target=_stream_inference, daemon=True)
       └─ _send_piece(token) → reactor.callFromThread(caller.msg, piece)
            └─ (main thread) caller.msg(piece) 
       └─ Done → reactor.callFromThread(caller.msg, "⚡ 18 t/s")
```

### Key Decision
- **Threaded streaming is the right approach for Evennia** — Evennia's Twisted reactor is single-threaded. Blocking inference in the command handler means messages queue but don't flush until command returns. Spawning a background thread and using `reactor.callFromThread` for `caller.msg()` lets tokens appear progressively. The C mutex serializes concurrent access when multiple users call @infer.

### Summary

### Bugs Fixed

**Bug 1 — Undefined symbol `quantize_row_nvfp4_ref`**
- Root cause: When `edge-gateway.py` imports modules at startup, `ollama_client` transitively imports `llama_cpp`, which loads `libggml-cpu.so` with `RTLD_GLOBAL`. This library has `quantize_row_nvfp4_ref` as an **undefined weak symbol** — it's normally resolved when `libggml-cuda.so` initializes its CUDA device constructors. But with `CUDA_VISIBLE_DEVICES=""`, those constructors never run, leaving the symbol unresolved.
- When `NativeInference.load()` then calls `ctypes.cdll.LoadLibrary("libedge-cuda.so")`, the linker resolves `libllama.so` → `libggml-cpu.so` → already-loaded symbols → MISSING `quantize_row_nvfp4_ref` → dynamic linker error.
- **Fix**: Pre-load `libggml-cuda.so` *before* loading `libedge-cuda.so`, using `RTLD_GLOBAL` to register its symbols. This ensures `quantize_row_nvfp4_ref` (defined in the CUDA backend's matmul implementation) is available on the global symbol table before libllama tries to resolve it.
```python
_lib = ctypes.CDLL(lib_path, mode=ctypes.RTLD_GLOBAL)
# ... register function signatures, set restype/argtypes ...
# Pre-load CUDA lib to register weak symbols
if not hasattr(NativeInference, '_ggml_cuda_preloaded'):
    import glob as _glob
    _ggml_libs = _glob.glob(
        os.path.expanduser("~/.local/lib/python3.10/site-packages/llama_cpp/lib/libggml-cuda*.so"))
    if _ggml_libs:
        ctypes.cdll.LoadLibrary(_ggml_libs[0])
        NativeInference._ggml_cuda_preloaded = True
```

**Bug 2 — `backend_name` property broken**
- `edge_backend` returns `c_char_p` (Python `bytes`), but `backend_name` was calling `ctypes.string_at(ptr).decode()` on bytes, which passes the bytes to `string_at` as a pointer, causing garbage.
- **Fix**: Return `ptr.decode('utf-8')` directly since `c_char_p` already returns bytes.
```python
@property
def backend_name(self) -> str:
    if not self._impl:
        return "none"
    try:
        ptr = self._lib.edge_backend(self._impl)
        return ptr.decode('utf-8') if ptr else "unknown"
    except Exception:
        return "unknown"
```

**Bug 3 — `_native_chat_fallback` backend name check**
- The fallback handler checked `native_result["backend"] == "native"` but the real backend is `"llama.cpp (CPU)"`.
- **Fix**: Changed the check to accept all non-`"none"` backends: `if native_result["backend"] != "none":`
