## Executive Summary

**The answer is: one universal bootloader + role-specific bytecode programs, NOT one universal firmware.**

The ESP32-S3 hardware forces real divergence in interrupt handling, DMA allocation, and memory bus arbitration. But the *architecture* can be 90% shared — the divergence is in configuration, not code. An edge agent boards with a universal base, reads a capability descriptor, and loads the appropriate bytecode program for its role.

The key insight from all models: **roles A (motor) and C (camera/sensor) CANNOT coexist on the same ESP32** due to DMA channel contention (6 total, camera needs 4-5) and PSRAM bus saturation (160MB/s bursts stall motor ISRs). But B (monitor) and D (interface) can coexist with either A or C.

## 1. Hardware Constraints That Force Divergence (DeepSeek-chat + DeepSeek-Reasoner)

### ESP32-S3 Resource Inventory
- Dual-core Xtensa LX7 @ 240MHz
- 512KB SRAM, up to 8MB PSRAM (80MHz QSPI)
- 6 GDMA channels (TX + RX)
- 4 hardware timers
- WiFi/BLE stack locked to Core 0
- Interrupt priority levels 1-3 (1 = highest)

### Critical Conflicts

| Resource | Motor Control | Sensor Pipeline | Conflict? |
|----------|--------------|-----------------|-----------|
| DMA channels | 1-2 (PWM, ADC) | 4-5 (camera, I2S) | **YES** (6 total) |
| PSRAM bus | Minimal (SRAM-only) | 160MB/s bursts | **YES** (bus saturation) |
| Interrupt latency | <10µs consistent | 2-5µs jitter from VSYNC | **YES** |
| Core affinity | Core 1 (away from WiFi) | Core 0 (camera driver) | **NO** (separate cores) |
| GPIO pins | Direction + encoder | Camera 8-16 pins | **NO** (different pins) |
| Timers | 1-2 dedicated (PID) | Optional | **MAYBE** (4 total) |

### The Definitive Answer: Can Motor + Camera Coexist?
**No, not reliably.** The DMA contention and PSRAM bus saturation guarantee motor ISR deadline misses during frame capture.

**Workaround if forced:** Camera at 1-5 FPS (reduces bus contention), motor PWM generated by hardware timer (not DMA), motor ISR at priority level 1.

### Core Pinning Requirements
- **Motor Control (A):** Core 1, priority 1, WiFi disabled on Core 1
- **Sensor Pipeline (C):** Core 0, priority 2-3, DMA-heavy
- **Deep Monitor (B):** Either core, priority 2, low compute
- **Control Interface (D):** Either core, priority 3, human-speed

## 2. The Universal Edge Agent Architecture (Kimi K2.5)

### Vessel Descriptor: Capability Contracts, Not Config Files

The descriptor is a **Merkleized DAG of Abstract Resource IDs (ARIDs)**, not pin mappings. An ARID is a function (`STERN_THRUSTER_PWM`) with timing constraints (deadband: 50µs, fail-safe: 1500µs) and electrical profiles (open-drain, 400kHz max).

**Descriptor contents:**
- **Hardware Signatures:** SHA-256 hashes of I2C device ROMs and CAN UUIDs — distinguishes "absent" from "replaced with hostile clone"
- **Fail-Safe Hypercube:** State-space matrix defining safe outputs for every input vector when comms drop
- **Trust Lattice ACL:** Which ARIDs require which INCREMENTS level. L0 reads `BATTERY_VOLTAGE`; L3 writes `MAIN_BUS_CONTACTOR`

**Security:** Descriptor arrives over untrusted channel (BLE/UDP). Parsed by constant-time decoder — two-pass: (1) validate signatures, expand ARIDs to 256-entry lookup table in RTC fast memory, (2) bind physical pins.

### Hardware Discovery: Conservative Probing Protocol

1. **High-Impedance Recon:** All GPIOs start as inputs, pull-ups disabled. ULP coprocessor samples for 100ms. Lines at 5V flagged as hazardous.
2. **Progressive Capability Claim:** Agent claims ARIDs one at a time, testing each against descriptor expectations before enabling output drivers.
3. **Signature Verification:** I2C/CAN devices verified against SHA-256 hashes before trust is established.

### Role Switching Mid-Operation

- Bytecode VM can hot-swap programs via `LOAD_PROGRAM` opcode
- Trust resets to L0 for new role, requires re-accumulation
- Hardware reconfiguration requires safe-state transition: all outputs to fail-safe positions → rebind pins → load new program → increment trust from L0

## 3. Industry Reference Architectures (DeepSeek-chat)

### Marine Autopilots
- **Raymarine/Simrad/B&G:** Separate processors for display (ARM), autopilot control (dedicated MCU), and sensor fusion. Connected via NMEA 2000 (CAN 2.0B at 250Kbaud) and proprietary protocols (Seatalk NG = NMEA 2000).
- **Key insight:** Marine industry already separates safety-critical motor control from navigation. This is not optional — it's regulatory (ISO 11562 for autopilots).

### Industrial Robotics
- **KUKA/Fanuc:** Dedicated safety PLC (separate MCU) watches the main controller. Safety PLC can cut power independently. Controller runs motion planning; PLC runs safety monitoring.
- **Key insight:** Physical separation of control and monitoring is industry standard. Software separation is considered insufficient.

### Drone Flight Controllers
- **Pixhawk/ArduPilot:** Single STM32H7 for real-time (400Hz-8kHz motor loop) + companion computer (Raspberry Pi/Jetson) for vision/navigation. The STM32 runs a cooperative scheduler — NO FreeRTOS, just bare-metal interrupt-driven loop.
- **Key insight:** For hard real-time, bare-metal beats RTOS. The NEXUS bytecode VM running on bare metal (no OS) would match this pattern.

### Tesla Autopilot
- **Hardware 3/4:** Custom neural accelerator (NPU) for inference + ARM Cortex for planning + dedicated safety MCU (Infineon AURIX, ASIL-D rated) for final command verification. Three physically separate silicon dies.
- **Key insight:** Tesla uses a dedicated ASIL-D safety chip that can independently cut power. This is the extreme version of the "deep monitor" role.

## 4. Edge Equipment Catalog (DeepSeek-chat)

### Selection Matrix

| Equipment | Hardware | Trust Min | Flash | RAM | PSRAM | Shares With |
|-----------|----------|-----------|-------|-----|-------|-------------|
| **motor-control** | 2x PWM, 2x encoder, 2x ADC, 1x timer | L3 | 48KB | 8KB | None | sensor, comms |
| **deep-monitor** | 1x CAN, 1x UART | L2 | 64KB | 12KB | 256KB | comms, sensor |
| **sensor-pipeline** | 2x I2C, 1x I2S, 1x SPI, 4x DMA | L2 | 96KB | 16KB | 4MB | comms |
| **nav-compute** | 1x UART (GPS), 1x I2C (IMU) | L2 | 48KB | 12KB | 512KB | monitor, comms |
| **marine-bridge** | 1x CAN, 1x UART, 1x NMEA decoder | L2 | 32KB | 8KB | 64KB | monitor, interface |
| **comms-relay** | WiFi + BLE + 1x UART | L1 | 48KB | 12KB | 128KB | all (non-motor) |

### Compatible Combinations (per ESP32)
- ✅ **motor-control + comms-relay** (different peripherals, different cores)
- ✅ **deep-monitor + marine-bridge** (both CAN users, can share bus)
- ✅ **sensor-pipeline + nav-compute** (I2C shared, DMA allocated separately)
- ✅ **deep-monitor + control-interface** (low compute, plenty of headroom)
- ❌ **motor-control + sensor-pipeline** (DMA + PSRAM bus conflict)
- ❌ **motor-control + motor-control** (timer + DMA conflict)
- ⚠️ **sensor-pipeline + comms-relay** (WiFi ISRs on Core 0 may affect DMA — possible with care)

## 5. The Answer: How to Build This

### Architecture Decision: Capability-Descriptor + Role Bytecode

**One universal base firmware (~128KB flash) that:**
1. Boots in L0 probe mode (high-impedance GPIO, ULP reconnaissance)
2. Reads vessel descriptor from I2C EEPROM / CAN broadcast / GPIO strap
3. Validates descriptor signatures (Merkle DAG, SHA-256)
4. Binds hardware: ARID → physical pin/peripheral mapping
5. Loads role-specific bytecode program from flash or over-the-air
6. Enters INCREMENTS trust accumulation (L0 → role's minimum)
7. Reports capabilities to fleet via Jetson Orin

**Role-specific bytecode programs (~16-48KB each):**
- `motor_ctl.byc` — PID loop, PWM generation, encoder reading, current limiting
- `deep_mon.byc` — CAN snooping, anomaly scoring, Jetson alert protocol
- `sensor_pipe.byc` — I2C/I2S/SPI polling, DMA configuration, ESP-DSP inference
- `ctrl_iface.byc` — GPIO polling, CAN bridge, display SPI, USB configuration
- `nav_comp.byc` — GPS/IMU fusion, waypoint following, danger zone detection
- `comms_relay.byc` — WiFi/BLE/UART bridge, message routing, fleet protocol

### Why This Works
- **90% code shared** (bootloader, comms, trust, OTA, watchdog, logging)
- **10% role-specific** (bytecode program + hardware binding)
- **Hardware-agnostic** at the agent level — descriptor tells it what's connected
- **Trust-gated** — dangerous roles (motor) require L3; monitoring starts at L2
- **Fleet-compatible** — any vessel can equip any equipment by loading the right bytecode

### What SuperInstance Should Build First
1. **Universal bootloader with probe mode** (~2 weeks)
2. **Capability descriptor format** (JSON schema + Merkle signing) (~1 week)
3. **motor_ctl.byc** bytecode program (highest safety, most constrained) (~3 weeks)
4. **deep_mon.byc** bytecode program (most novel — anomaly detection) (~2 weeks)
5. **Integration test: motor ESP32 + monitor ESP32 + Jetson** (~2 weeks)

### The Big Insight

> **The difference between roles is NOT firmware architecture — it's INTERRUPT PRIORITY and DMA ALLOCATION.** The bytecode VM, trust system, comms stack, and OTA mechanism are identical across all roles. What changes is: which core runs the main loop, what priority the ISRs get, and which DMA channels are allocated.

This means SuperInstance's massive Edge-Native knowledge base CAN be decomposed into focused repo-agents. The universal base goes in one repo. Each equipment type (motor, monitor, sensor, nav, bridge, comms) goes in its own repo as a bytecode program. The vessel builder picks their hardware, loads the universal base on each ESP32, and equips each unit with the appropriate bytecode program.

**6 specialized repo-agents + 1 universal base = the edge fleet.**
